## 1. AI 引擎选型

### 1.1 核心需求

雅思学习平台需要 AI 完成以下任务：
- **写作批改**：按雅思四项评分标准（Task Response / Coherence / Lexical Resource / Grammar）打分 + 逐句批注
- **口语模拟**：模拟考官对话，评估流利度、词汇、语法、发音
- **阅读出题**：根据文章生成选择题、判断题、填空题
- **听力出题**：根据音频或文本生成听力题目

### 1.2 免费 API 对比

| 平台 | 模型 | 免费额度 | 国内直连 | 适合场景 |
|------|------|----------|----------|----------|
| **DeepSeek** | deepseek-chat (V3) | 注册送 500 万 token，之后按量付费（极便宜） | ✅ | 写作批改、阅读出题 |
| **阿里通义千问** | qwen-turbo / qwen-plus | 百万 token 免费额度 | ✅ | 通用对话、题目生成 |
| **智谱 GLM** | glm-4-flash | 永久免费（限速） | ✅ | 写作批改 |
| **Groq** | llama-3.3-70b 等 | 免费（速率限制） | ⚠️ 需代理 | 高性能推理 |
| **OpenAI** | gpt-4o-mini | 付费，约 $0.15/百万 token | ❌ 需代理 | 效果最好 |
| **Google Gemini** | gemini-1.5-flash | 免费 1500 请求/天 | ❌ 需代理 | 多模态（图片题） |

### 1.3 推荐方案

**首选：DeepSeek**

理由：
- 国内直接访问，无需代理
- 中文理解能力强，适合中国考生
- API 兼容 OpenAI 格式，换模型零成本
- 价格极低（百万 token 约 ¥1）

**备选：通义千问（阿里云）**

理由：
- 国内生态完善，有免费额度
- 如果后续想扩展语音功能，阿里有配套 TTS/ASR

**架构建议：做一个「模型适配层」**

```python
# 统一接口，随时切换 AI 提供商
class AIProvider:
    def chat(self, messages: list) -> str: ...
    def score_essay(self, essay: str) -> dict: ...

class DeepSeekProvider(AIProvider): ...
class QwenProvider(AIProvider): ...
```

这样即使将来换模型，只改一行配置。

### 1.4 AI 评分准确性

根据 2026 年 Nature 论文《Enhancing IELTS writing automated scoring with M-LoRA fine-tuned LLAMA-3》：

- 经过微调的 LLM 雅思作文评分与真人考官相关系数达到 **0.85+**
- 直接使用通用大模型（不微调）通过精心设计的 prompt 也能达到 **0.75-0.82** 的准确度
- **结论**：通用大模型 + 好的 prompt 工程已经足够实用，不需要自己训练模型

### 1.5 Prompt 设计要点

让 AI 按雅思官方标准评分的关键在于 prompt：

```
你是一名雅思写作考官，请按以下四项标准评分（每项 0-9 分）：

1. Task Response（任务完成度）
2. Coherence and Cohesion（连贯与衔接）
3. Lexical Resource（词汇资源）
4. Grammatical Range and Accuracy（语法范围与准确性）

请对以下作文评分，并逐项给出：
- 得分（含 0.5 档）
- 优点（至少 2 条）
- 待改进（至少 2 条）
- 修改建议（具体到句子）
```

### 1.6 成本估算

| 操作 | Token 消耗 | DeepSeek 费用 |
|------|-----------|--------------|
| 批改一篇 250 词作文 | ~2000 token | ¥0.002 |
| 生成一篇阅读 + 题目 | ~3000 token | ¥0.003 |
| 一次口语对话（10 轮） | ~5000 token | ¥0.005 |
| **日均 100 次使用** | ~300k token | **¥0.30** |

> 结论：用 DeepSeek，API 成本基本可以忽略不计。

## 2. Python Web 框架对比

### 2.1 三大选择

```
难度：  Flask < FastAPI < Django
性能：  Django < Flask < FastAPI
生态：  Django > Flask > FastAPI
新手友好：Flask > FastAPI > Django
```

| 特性 | Flask | FastAPI | Gradio |
|------|-------|---------|--------|
| 学习曲线 | ⭐ 简单 | ⭐⭐ 中等 | ⭐ 极简 |
| 性能 | 中等 | 高（异步） | 低 |
| 前端需求 | 需要 HTML/CSS/JS | 需要 HTML/CSS/JS | **不需要** |
| API 文档 | 需手动/插件 | **自动生成 Swagger** | 自动生成 |
| 适合场景 | 小项目/原型 | 中大型/API 驱动 | 快速原型/AI demo |
| 数据库集成 | 手动 | SQLAlchemy/SQLModel | 不支持 |
| 用户系统 | Flask-Login 等 | FastAPI-Users 等 | 自带简易认证 |

### 2.2 分阶段推荐

```
阶段 1（原型验证）: Gradio
  - 一小时搭出写作批改 demo
  - 不用写任何 HTML/CSS/JS
  - 确认 AI 评分效果

阶段 2（功能完善）: FastAPI
  - 当 Gradio 不够用（需要自定义 UI、数据库、用户系统）
  - 迁到 FastAPI + Jinja2 模板 或 FastAPI + React/Vue

阶段 3（正式产品）: FastAPI + 前端框架
  - 前后端分离
  - 移动端适配
```

### 2.3 FastAPI 为什么是最佳长期选择

1. **自动生成 API 文档**：访问 `/docs` 就看到所有接口，方便调试
2. **类型校验**：Python 类型提示自动做输入验证，减少 bug
3. **异步支持**：AI API 调用是 I/O 密集型，异步能大幅提升性能
4. **生态成熟**：SQLModel、FastAPI-Users、FastAPI-Cache 等即插即用

```

MVP 阶段 → FastAPI + Jinja2 模板（传统服务端渲染）
成品阶段 → FastAPI（纯 API）+ Vue.js SPA
```

### 3.3 Jinja2 模板方案（对新手最友好）

```python
# 后端路由

@app.post("/writing/score")

    result = ai.score_essay(essay)
    return templates.TemplateResponse("result.html", {"result": result})
```

```html
    <button type="submit">提交批改</button>

```

## 7. 项目架构设计

### 7.1 什么是 Web 应用？

```
用户浏览器 → [前端页面] → HTTP 请求 → [后端 Python] → [AI API]
                  ↑                        ↓
                  ←─────── HTTP 响应 ──────┘
```

**一句话解释**：用户在网页上操作 → 浏览器发请求给 Python 程序 → Python 处理后返回结果 → 浏览器显示结果。

### 7.2 项目目录结构

```
ielts-app/
├── app/
│   ├── __init__.py
│   ├── main.py              # FastAPI 入口、路由注册
│   ├── config.py             # 配置文件（API key 等）
│   ├── routers/              # 路由（每个模块一个文件）
│   │   ├── writing.py        # 写作模块路由
│   │   ├── speaking.py       # 口语模块路由
│   │   ├── reading.py        # 阅读模块路由
│   │   └── listening.py      # 听力模块路由
│   ├── services/             # 业务逻辑（与路由解耦）
│   │   ├── ai_provider.py    # AI 模型适配层
│   │   ├── writing_scorer.py # 写作评分逻辑
│   │   ├── question_gen.py   # 题目生成逻辑
│   │   └── tts_service.py    # 语音合成
│   ├── models/               # 数据库模型
│   │   ├── user.py
│   │   ├── writing.py
│   │   └── ...
│   ├── schemas/              # Pydantic 数据校验
│   │   ├── writing.py
│   │   └── ...
│   └── templates/            # Jinja2 HTML 模板
│       ├── base.html         # 基础布局
│       ├── writing.html
│       └── ...
├── static/                   # CSS、JS、图片
├── tests/                    # 测试
├── requirements.txt
├── .env                      # 环境变量（不提交到 git）
└── .gitignore
```

### 7.3 关键概念解释

| 概念 | 白话解释 | 对应文件 |
|------|----------|----------|
| **路由 (Router)** | "用户访问 /writing 时，谁来处理？" | `routers/writing.py` |
| **服务 (Service)** | 核心业务逻辑，比如"怎么给作文打分" | `services/writing_scorer.py` |
| **模型 (Model)** | 数据库表的结构定义 | `models/writing.py` |
| **Schema** | 前端传过来的数据长什么样、合不合法 | `schemas/writing.py` |
| **模板 (Template)** | 网页的 HTML 骨架 | `templates/writing.html` |

### 7.4 请求流程示例（写作批改）

```
1. 用户在 writing.html 粘贴作文 → 点提交
2. 浏览器 POST /writing/score?essay=...
3. routers/writing.py 收到请求
4. 用 schemas/writing.py 校验数据
5. 调用 services/writing_scorer.py 评分
6. writing_scorer.py 调用 services/ai_provider.py → DeepSeek API
7. 等 AI 返回结果
8. 结果存入 models/writing.py → SQLite
9. 渲染 result.html → 返回给浏览器
```

## 8. 安全性

### 8.1 常见安全风险与对策

| 风险 | 说明 | 对策 | 难度 |
|------|------|------|------|
| **SQL 注入** | 恶意输入操作数据库 | 用 ORM（SQLAlchemy），不用拼接 SQL | 几乎零成本 |
| **XSS 跨站脚本** | 用户输入 <script> 恶意代码 | Jinja2 默认转义 HTML | 几乎零成本 |
| **CSRF 攻击** | 伪造用户请求 | FastAPI 不支持 Cookie Session，天然免疫大部分 | 低 |
| **API Key 泄露** | 代码中硬编码密钥 | 用 `.env` 文件 + 环境变量，`.gitignore` | 极低 |
| **密码明文存储** | 数据库被盗 → 密码泄露 | bcrypt 哈希存储 | 极低 |
| **暴力破解** | 反复试密码 | 登录限速（5 次/分钟） | 低 |
| **文件上传漏洞** | 上传恶意文件 | 限制格式、大小，不上传到 Web 目录 | 中 |
| **AI Prompt 注入** | 用户输入诱导 AI 做其他事 | 输入长度限制 + 服务端 prompt 包裹 | 低 |

### 8.2 安全实现示例

```python
# 1. API Key 保护
# .env 文件（不提交 git）
DEEPSEEK_API_KEY=sk-xxxxx

# config.py
import os
from dotenv import load_dotenv
load_dotenv()
API_KEY = os.getenv("DEEPSEEK_API_KEY")  # 不从代码里读

# 2. 密码哈希
from passlib.hash import bcrypt
password_hash = bcrypt.hash("用户密码")  # 存这个，不存明文

# 3. 输入限制
MAX_ESSAY_LENGTH = 5000  # 最多 5000 字符
if len(essay) > MAX_ESSAY_LENGTH:
    raise HTTPException(400, "作文超出长度限制")

# 4. 登录限速
from slowapi import Limiter
limiter = Limiter(key_func=get_remote_address)

@app.post("/login")
@limiter.limit("5/minute")  # 每分钟最多 5 次
def login(): ...
```

### 8.3 安全优先级

```
必须做的（不做就出事）:
  ✅ API Key 不提交 git
  ✅ 密码哈希存储
  ✅ 输入长度限制

建议做的（花半小时，安心很久）:
  ✅ SQLAlchemy ORM（防注入）
  ✅ HTTPS（Render 免费提供）
  ✅ 登录限速

可以以后做的（有用户了再说）:
  ⏸ 验证码
  ⏸ 双因素认证
  ⏸ 安全审计日志
```

## 9. 模块解耦与可维护性

### 9.1 什么是解耦？

**耦合（坏）**：改写作模块的代码，口语模块崩了。

```
❌ 耦合写法：
def score_writing(essay):
    result = requests.post("https://api.deepseek.com/...", ...)
    db.execute("INSERT INTO writings ...")
    render_template("result.html", result)

# 问题：API 换了要改这里，数据库换了要改这里，模板换了要改这里
```

**解耦（好）**：每层各管各的，互不影响。

```
✅ 解耦写法：
# 路由层 —— 只管「谁处理这个 URL」
@app.post("/writing/score")
def score(essay: str):
    result = writing_service.score(essay)
    return result

# 服务层 —— 只管「怎么评分」
class WritingService:
    def score(self, essay):
        prompt = build_prompt(essay)
        raw = ai.chat(prompt)
        return parse_result(raw)

# AI 层 —— 只管「跟哪个 AI 说话」
class AIProvider:
    def chat(self, messages): ...
```

### 9.2 四个模块如何解耦

```
                    ┌──────────────┐
                    │  main.py     │  ← 注册所有路由
                    └──────┬───────┘
           ┌───────────────┼───────────────┐
    ┌──────▼──────┐ ┌─────▼──────┐ ┌──────▼──────┐
    │ writing     │ │ speaking   │ │ reading     │ ...  ← 各自独立路由
    │ router      │ │ router     │ │ router      │
    └──────┬──────┘ └─────┬──────┘ └──────┬──────┘
    ┌──────▼──────┐ ┌─────▼──────┐ ┌──────▼──────┐
    │ writing     │ │ speaking   │ │ question    │      ← 各自独立服务
    │ service     │ │ service    │ │ service     │
    └──────┬──────┘ └─────┬──────┘ └──────┬──────┘
           └───────────────┼───────────────┘
                    ┌──────▼──────┐
                    │ AI Provider │     ← 共享底层（唯一共享的东西）
                    └─────────────┘
```

> 每个模块的 router、service、model 各自独立，只有 `AI Provider` 是公共的。
> 想加第五个模块？新建文件夹，写 router + service，main.py 里注册一行，完事。

## 11. 总结与推荐方案

### 11.1 推荐技术栈

```
后端框架:    FastAPI
前端:        第一阶段 Gradio → 第二阶段 Jinja2 模板
AI 引擎:     DeepSeek（主）+ 通义千问（备）
数据库:      SQLite → PostgreSQL（未来）
部署:        第一阶段 Render（免费）→ 第二阶段阿里云
版本控制:    Git + GitHub
```

### 11.2 分阶段路线图

```
Phase 0 — 原型验证（1-2 天）
  ├── Gradio 写一个写作批改 demo
  ├── 接入 DeepSeek API
  ├── 验证 AI 评分效果
  └── 决定要不要继续

Phase 1 — 写作模块 MVP（1-2 周）
  ├── 切换到 FastAPI + Jinja2
  ├── 写作批改完整流程
  ├── 用户注册/登录
  ├── 历史记录查看
  └── 部署到 Render

Phase 2 — 口语模块（1-2 周）
  ├── AI 考官对话
  ├── 话题卡生成
  └── 口语评估反馈

Phase 3 — 阅读模块（1-2 周）
  ├── AI 生成文章 + 题目
  ├── 自动对答案
  └── 错题解析

Phase 4 — 听力模块（1-2 周）
  ├── AI 生成听力材料
  ├── TTS 朗读
  └── 题目 + 答案

Phase 5 — 打磨上线（持续）
  ├── 界面优化
  ├── 性能优化
  ├── 国内服务器部署
  └── 用户反馈迭代
```

### 11.3 核心原则

1. **先跑通再优化** — 原型用 Gradio，不纠结架构
2. **一次只做一个模块** — 写作模块没做好之前，不开口语
3. **免费优先** — API 用免费的，部署用免费的，不加服务器
4. **面向中国考生** — 中文界面、中文反馈、国内能访问
