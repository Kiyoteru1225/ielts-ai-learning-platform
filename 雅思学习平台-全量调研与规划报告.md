## 一、项目概述

### 1.1 一句话定位

为中国雅思考生提供 AI 驱动的听说读写全模块学习平台（Web 应用）。

### 1.2 目标用户

准备雅思考试的中国学生，英语水平在 4-6 级之间，使用手机或电脑访问。

### 1.3 核心价值

| 痛点 | 解决方案 |
|------|----------|
| 写作没人批改 | AI 按雅思四项标准打分 + 逐句建议 |
| 口语没有陪练 | AI 模拟考官对话，随时可练 |
| 真题不够刷 | AI 生成海量题目，无限供应 |
| 外教太贵 | 完全免费或极低成本 |

### 1.4 功能优先级

```
P0 — 第一版必须有（核心闭环）
  ✅ 写作批改：粘贴作文 → AI 评分 + 逐项反馈
  ✅ 用户注册/登录
  ✅ 历史记录查看

P1 — 第二版（扩大覆盖面）
  ⬜ 口语模拟：AI 考官对话 + 评分反馈
  ⬜ 词汇模块：高频词库 + 间隔复习 + AI 出题
  ⬜ 阅读练习：AI 生成文章 + 题目 + 自动对答案

P2 — 第三版（锦上添花）
  ⬜ 听力练习：AI 生成对话 + TTS 朗读
  ⬜ 错题本 + 学习统计
  ⬜ 模拟考试模式（计时 + 限时）

不做：
  ❌ 不接入真题（版权风险）
  ❌ 不做手机 App（先用 Web 响应式）
  ❌ 不做社交功能（评论、点赞、社区）
```

## 五、AI 引擎选型

### 5.1 核心需求

雅思学习平台需要 AI 完成：
- **写作批改**：按雅思四项评分标准打分 + 逐句批注
- **口语模拟**：模拟考官对话，评估回答质量
- **阅读出题**：生成文章 + 选择题/判断题/填空题 + 答案
- **听力出题**：生成对话脚本 + TTS 朗读 + 题目

### 5.2 免费 API 对比

| 平台 | 模型 | 免费额度 | 国内直连 | 推荐场景 |
|------|------|----------|----------|----------|
| **DeepSeek** | deepseek-chat (V3) | 注册送 500 万 token | ✅ | 写作、出题 **← 首选** |
| 阿里通义千问 | qwen-turbo/plus | 百万 token 免费 | ✅ | 通用对话、TTS 配套 |
| 智谱 GLM | glm-4-flash | 永久免费（限速） | ✅ | 写作批改 |
| Groq | llama-3.3-70b | 免费（限速） | ⚠️ 需代理 | 高性能推理 |
| OpenAI | gpt-4o-mini | 付费 | ❌ 需代理 | 效果最好 |
| Google Gemini | gemini-1.5-flash | 1500 请求/天 | ❌ 需代理 | 多模态 |

### 5.3 推荐：DeepSeek（主）+ 通义千问（备）

**DeepSeek 理由：**
- 国内直连，免代理
- 中文强，适合中国考生
- API 兼容 OpenAI 格式，切模型零成本
- 极便宜（百万 token ≈ ¥1）

**架构设计：模型适配层**

```python
# 统一接口，随时切换
class AIProvider:
    def chat(self, messages: list[dict]) -> str: ...
    def score_essay(self, essay: str) -> dict: ...

class DeepSeekProvider(AIProvider): ...
class QwenProvider(AIProvider): ...

# 使用时
ai = AIProvider.create("deepseek")  # 改一行切换
```

### 5.4 AI 评分准确性

2026 年 Nature 论文结论：
- 微调 LLM 评分与真人考官相关系数 **0.85+**
- 通用大模型 + 精心 prompt **0.75-0.82**
- **结论：不训练模型，靠 prompt 工程就能用**

### 5.5 评分 Prompt 设计

```
你是一名雅思写作考官，请按以下四项标准评分（0-9 分）：

1. Task Response（任务完成度）
2. Coherence and Cohesion（连贯与衔接）
3. Lexical Resource（词汇资源）
4. Grammatical Range and Accuracy（语法准确性）

对以下作文给出：
- 每项得分（含 0.5 档）+ 总分
- 优点（≥2 条）
- 待改进（≥2 条）
- 具体到句子的修改建议
```

### 5.6 成本估算

| 操作 | Token 消耗 | DeepSeek 费用 |
|------|-----------|--------------|
| 批改一篇 250 词作文 | ~2,000 | ¥0.002 |
| 生成一篇阅读 + 13 题 | ~3,000 | ¥0.003 |
| 一次口语对话（10 轮） | ~5,000 | ¥0.005 |
| **日均 100 次使用** | ~300,000 | **¥0.30** |

## 六、Python Web 框架对比

```
难度：     Flask < FastAPI < Django
性能：     Django < Flask < FastAPI
生态：     Django > Flask > FastAPI
新手友好： Flask > FastAPI > Django
```

| 特性 | Flask | FastAPI | Gradio |
|------|-------|---------|--------|
| 学习曲线 | ⭐ 简单 | ⭐⭐ 中等 | ⭐ 极简 |
| 性能 | 中 | **高（异步）** | 低 |
| 前端需求 | 需 HTML/CSS/JS | 需 HTML/CSS/JS | **不需要** |
| API 文档 | 手动 | **自动 Swagger** | 自动 |
| 数据库集成 | 手动 | SQLAlchemy/SQLModel | 不支持 |
| 生产就绪 | 需插件 | ✅ | ❌ |

### 分阶段选择

```
阶段 1（原型） → Gradio   （纯 Python，1 小时跑通）
阶段 2（MVP）  → FastAPI + Jinja2（服务端渲染）
阶段 3（成品） → FastAPI + Vue.js（前后端分离）
```

### FastAPI 长期优势

1. `/docs` 自动生成 API 文档，调试零成本
2. Python 类型提示自动做输入校验
3. 异步原生支持（AI API 调用是 I/O 密集型）
4. 生态成熟：SQLModel、FastAPI-Users、FastAPI-Cache

### 推荐：Jinja2 模板

不学 JS 框架也能做出像样的页面：

```python
# 后端路由（Python-only）
@app.get("/writing")
def writing_page(request: Request):
    return templates.TemplateResponse("writing.html", {"request": request})

@app.post("/writing/score")
def score_essay(essay: str = Form(...)):
    result = ai.score_essay(essay)
    return templates.TemplateResponse("result.html", {"result": result})
```

```html
<!-- 前端模板：会 HTML 就能写 -->
<form method="post" action="/writing/score">
    <textarea name="essay" rows="15"
        placeholder="在此粘贴你的雅思作文..."></textarea>
    <button type="submit">提交批改</button>
</form>
```

配合 Bootstrap CSS，直接有好看的按钮、表单、卡片。不需要写一行 CSS。

### 核心表设计

```sql
-- 用户
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    username TEXT UNIQUE NOT NULL,
    email TEXT UNIQUE,
    password_hash TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 写作记录
CREATE TABLE writing_records (
    id INTEGER PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    task_type TEXT NOT NULL,    -- 'task1' / 'task2'
    essay_text TEXT NOT NULL,
    score_tr REAL,              -- Task Response
    score_cc REAL,              -- Coherence & Cohesion
    score_lr REAL,              -- Lexical Resource
    score_gra REAL,             -- Grammatical Range
    feedback_json TEXT,         -- AI 完整返回
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 口语记录、阅读记录、听力记录（同理）
```

## 二十、AI 异常处理

AI 一定会出问题，不是概率问题，是时间问题。

| 情况 | 表现 | 对策 |
|------|------|------|
| API 超时 | DeepSeek 高峰期 30 秒没响应 | timeout 30s + 重试 2 次 |
| 返回格式错 | 让它回 JSON，回了纯文本 | try/except 兜底，给默认值 |
| AI 胡扯 | 给优秀作文打 3 分 | 分数范围校验，异常弹警告 |
| API 欠费 | 额度用完 | 前端显示"服务暂不可用" |

### 健壮调用示例

```python
import asyncio

async def safe_score_essay(essay: str, retries: int = 2) -> dict:
    for attempt in range(retries + 1):
        try:
            result = await asyncio.wait_for(
                ai.chat(build_prompt(essay)),
                timeout=30.0
            )
            parsed = parse_score(result)
            # 分数合理性检查
            if not (0 <= parsed["overall"] <= 9):
                raise ValueError("Invalid score")
            return parsed
        except asyncio.TimeoutError:
            if attempt == retries:
                raise HTTPException(504, "AI 服务响应超时，请稍后重试")
        except (ParseError, ValueError):
            if attempt == retries:
                raise HTTPException(502, "AI 返回格式异常，请重试")

    raise HTTPException(500, "未知错误")
```

## 二十三、成本监控与数据安全

### 23.1 API 用量追踪

免费 API 不等于不用管——你可能不知道哪天额度突然用完。

```python
import logging

class CostTracker:
    def __init__(self):
        self.total_tokens = 0
        self.total_cost = 0.0

    def track(self, prompt_tokens: int, completion_tokens: int):
        self.total_tokens += prompt_tokens + completion_tokens
        # DeepSeek 价格：输入 ¥1/百万 token，输出 ¥2/百万 token
        self.total_cost += (
            prompt_tokens * 0.000001 +
            completion_tokens * 0.000002
        )
        logging.info(f"累计: {self.total_tokens} tokens, ¥{self.total_cost:.4f}")

        if self.total_cost > 1.0:  # 每天超过 1 块钱
            logging.warning("⚠️ API 费用已超 ¥1/天")
```

## 二十七、中间件与横向扩展

当日活和访问量增长，单进程 Python 扛不住时，需要逐层加中间件扩容。

### 27.1 两种"中间件"

"中间件"在中国技术圈有两个含义，都要了解：

**A. FastAPI 中间件（请求拦截层）**

每个 HTTP 请求都先经过它，做统一的预处理。一行代码加一个。

```python
@app.middleware("http")
async def add_process_time(request: Request, call_next):
    start = time.time()
    response = await call_next(request)
    response.headers["X-Process-Time"] = str(time.time() - start)
    return response
```

**B. 基础设施中间件（扩展层）**

独立的服务进程：Redis、Nginx、Celery、消息队列等，用来分流、缓存、异步化。

### 27.2 扩展路线图

```
当前（1 个 FastAPI 进程）
  │
  │ 日活 > 50，AI 调用开始排队
  ▼
加 Redis
  ├── 缓存 AI 评分结果（同一篇作文不重复调 API）
  ├── Session 存 Redis（多进程共享）
  └── 限流计数器
  │
  │ 日活 > 200，CPU 打满
  ▼
加 Nginx（反向代理 + 负载均衡）
  ├── Nginx 在前接请求 → 分发给 3 个 FastAPI 进程
  ├── 静态文件（CSS/JS/图片/音频）Nginx 直接返回
  └── GZip 压缩交给 Nginx（比 Python 快 10 倍）
  │
  │ 日活 > 1000，数据库撑不住
  ▼
PostgreSQL + 读写分离
  ├── 主库写入 → 从库读取
  └── pgbouncer 连接池
  │
  │ 日活 > 5000
  ▼
Celery + RabbitMQ（异步任务队列）
  ├── AI 调用全部异步化
  └── 用户提交 → 进队列 → Worker 处理 → 通知用户
```

### 27.3 各中间件详解

#### Redis（缓存 + Session）

解决的问题：同一篇作文被多人提交时，每次都调 AI → 浪费钱和时间。

```python
import redis, json, hashlib

cache = redis.Redis(host='localhost', port=6379, decode_responses=True)

async def score_essay(essay: str) -> dict:
    essay_hash = hashlib.md5(essay.encode()).hexdigest()
    cache_key = f"essay:{essay_hash}"

    # 缓存命中 → 直接返回，不调 AI
    cached = cache.get(cache_key)
    if cached:
        return json.loads(cached)

    # 缓存未命中 → 调 AI → 存缓存
    result = await ai.score_essay(essay)
    cache.setex(cache_key, 3600, json.dumps(result))
    return result
```

| 作用 | 效果 |
|------|------|
| 缓存 AI 结果 | 同一篇作文不重复调 API，省 99% 成本 |
| Session 存 Redis | 多进程共享登录状态 |
| 限流计数器 | 比内存版更可靠 |

#### Nginx（反向代理）

解决的问题：一个 Python 进程只能用一个 CPU 核心，多个进程需要前面有人分发请求。

```nginx
# /etc/nginx/sites-available/ielts
server {
    listen 80;
    server_name 你的域名.com;

    # 静态文件 → Nginx 直接返回，不经过 Python
    location /static/ {
        alias /app/static/;
        expires 30d;         # 浏览器缓存 30 天
    }
    location /audio/ {
        alias /data/audio/;
    }

    # API 请求 → 转发给 FastAPI 集群
    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # GZip 压缩
    gzip on;
    gzip_types text/plain application/json text/css text/javascript;
}

# 3 个 FastAPI 进程，轮流转发
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}
```

```
用户请求 → Nginx → 轮询分发 → FastAPI-1 / FastAPI-2 / FastAPI-3
                             每个进程独立处理
```

> 启动多个 FastAPI 进程：`gunicorn app.main:app -w 3 -k uvicorn.workers.UvicornWorker`

#### Celery（异步任务）

解决的问题：AI 调用可能要 20-30 秒，用户阻塞等待体验极差。

```python
# tasks.py
from celery import Celery

celery_app = Celery('ielts', broker='redis://localhost:6379/0')

@celery_app.task
def score_essay_async(essay: str, user_id: int):
    """在后台 Worker 里慢慢处理"""
    result = ai.score_essay(essay)
    db.save_result(user_id, result)
    return result

# router.py — 用户只拿到一个任务 ID，不等结果
@app.post("/writing/score")
def submit_essay(essay: str, user_id: int):
    task = score_essay_async.delay(essay, user_id)
    return {
        "task_id": task.id,
        "status": "排队中，预计 30 秒完成",
        "check_url": f"/writing/result/{task.id}"
    }

# 前端轮询这个接口拿结果
@app.get("/writing/result/{task_id}")
def get_result(task_id: str):
    task = score_essay_async.AsyncResult(task_id)
    if task.ready():
        return {"status": "done", "result": task.result}
    return {"status": "processing"}
```

```
用户提交 → 返回 task_id → 前端轮询
              ↓
         任务进 Redis 队列
              ↓
         Celery Worker 慢慢处理
              ↓
         结果写入数据库
              ↓
         前端轮询拿到结果
```

### 27.4 为什么架构已经预留了扩展点

报告里设计的 router/service/model 分离结构，让加中间件几乎不影响已有代码：

```
加 Redis 缓存   → 在 service 层改，router 层不用动
加 Nginx        → 纯配置文件，代码零改动
加 Celery       → 新建 tasks.py，router 加一个返回 task_id 的分支
加读写分离      → 改 config.py 一行（database_url_readonly）
```

> 这叫「架构预留扩展点」。设计时就想好绳子挂哪里，但不提前挂。

### 27.5 加中间件的原则

```
不是「将来可能需要」就加，是「现在不加就崩了」才加。

判断标准：
  ✅ 用户投诉"太慢了"              → 加缓存
  ✅ AI 调用排队超过 30 秒          → 加 Celery
  ✅ CPU 长期 > 80%                → 加 Nginx + 多进程
  ✅ 数据库查询超过 1 秒            → 加索引 / 读写分离

  ❌ "将来万一火了"                 → 别加
  ❌ "大厂都这么干"                 → 别加
  ❌ "看起来很专业"                 → 别加
```
